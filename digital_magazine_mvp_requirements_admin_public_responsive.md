# 디지털 매거진 MVP 요구사항 정의서

## 1. 문서 개요

### 1.1 목적
본 문서는 디지털 매거진 MVP를 개발하기 위한 기능 요구사항을 정의한다.  
서비스는 **일반 사용자용 웹 화면**과 **관리자용 웹 화면**으로 분리되며, 관리자는 `/admin` 경로를 통해 접속한다.

### 1.2 범위
MVP 범위는 아래 두 영역을 포함한다.

- 일반 사용자 화면
  - 메인페이지
  - 매거진 상세/뷰어 페이지
- 관리자 화면 (`/admin`)
  - 매거진 생성
  - 드래프트 저장
  - 발행/비발행 관리
  - 페이지 이미지 업로드 및 순서 변경

### 1.3 MVP 목표
- 최신 매거진과 지난호 매거진을 웹에서 쉽게 탐색할 수 있어야 한다.
- 모든 페이지는 데스크톱, 태블릿, 모바일을 고려한 반응형 디자인으로 제공되어야 한다.
- 매거진은 책 형태의 좌우 2페이지 뷰로 열람할 수 있어야 한다.
- 관리자는 이미지 기반의 매거진을 손쉽게 생성/수정/발행할 수 있어야 한다.

---

## 2. 사용자 유형

### 2.1 일반 사용자
- 메인페이지에서 최신 매거진과 이전 매거진을 확인할 수 있다.
- 원하는 매거진을 선택해 상세 뷰어로 이동할 수 있다.
- 뷰어에서 슬라이드 또는 이전/다음 버튼으로 페이지를 넘길 수 있다.

### 2.2 관리자
- `/admin`으로 접속할 수 있다.
- 매거진을 생성/수정/삭제할 수 있다.
- 매거진 메타정보(번호, 제목)를 입력할 수 있다.
- 매거진 페이지 이미지를 업로드할 수 있다.
- 페이지 이미지 순서를 드래그앤드롭으로 변경할 수 있다.
- 매거진을 드래프트 상태로 저장할 수 있다.
- 매거진을 Publish / Unpublish 할 수 있다.

## 3. 라우팅 구조

## 3.1 일반 사용자 화면
- `/`
  - 메인페이지
- `/magazines/:id`
  - 매거진 상세/뷰어 페이지

## 3.2 관리자 화면
- `/admin`
  - 관리자 대시보드 또는 매거진 목록
- `/admin/magazines`
  - 매거진 목록
- `/admin/magazines/new`
  - 매거진 생성
- `/admin/magazines/:id/edit`
  - 매거진 수정

> 참고: MVP에서는 인증 방식이 별도 정의되지 않았더라도 `/admin` 이하 경로는 관리자만 접근 가능해야 한다.

## 4. 공통 데이터 구조

## 4.1 매거진(Magazine)
| 항목 | 타입 | 설명 |
|---|---|---|
| id | string / number | 매거진 고유 ID |
| issueNumber | string / number | 매거진 호수 |
| title | string | 매거진 제목 |
| status | enum | `draft`, `published`, `unpublished` |
| coverImageUrl | string | 대표 썸네일 이미지 |
| createdAt | datetime | 생성일시 |
| updatedAt | datetime | 수정일시 |
| publishedAt | datetime nullable | 발행일시 |

## 4.2 매거진 페이지(MagazinePage)
| 항목 | 타입 | 설명 |
|---|---|---|
| id | string / number | 페이지 고유 ID |
| magazineId | string / number | 소속 매거진 ID |
| pageNumber | number | 페이지 번호 |
| imageUrl | string | 페이지 이미지 URL |
| sortOrder | number | 정렬 순서 |
| createdAt | datetime | 생성일시 |

### 4.3 상태값 정의
- `draft`
  - 관리자만 볼 수 있는 임시 저장 상태
  - 일반 사용자 메인/상세에 노출되지 않음
- `published`
  - 일반 사용자에게 노출되는 상태
- `unpublished`
  - 과거 발행 이력이 있더라도 현재 일반 사용자에게 노출되지 않는 상태

---

## 4.4 반응형 디자인 공통 원칙
- 모든 페이지는 반응형을 고려하여 디자인 및 구현되어야 한다.
- 지원 기준 화면은 아래와 같다.
  - 모바일
  - 태블릿
  - 데스크톱
- 화면 너비에 따라 레이아웃, 여백, 이미지 크기, 버튼 크기, 네비게이션 방식이 자연스럽게 변경되어야 한다.
- 주요 CTA 버튼, 캐러셀 조작 요소, 페이지 이동 버튼은 터치 환경에서도 쉽게 조작 가능해야 한다.
- 텍스트, 썸네일, 버튼, 폼 요소는 작은 화면에서도 잘리지 않고 읽을 수 있어야 한다.
- 관리자 화면과 일반 사용자 화면 모두 동일하게 반응형 대응 범위에 포함된다.


## 5. 일반 사용자 화면 요구사항

## 5.1 메인페이지 (`/`)

### 5.1.1 최신 매거진 노출
- 메인페이지 최상단에 **가장 최근 Publish 된 디지털 잡지 1개**가 대표 영역으로 노출되어야 한다.
- 대표 영역에는 아래 정보가 포함되어야 한다.
  - 대표 썸네일(Thumbnail)
  - 매거진 호수
  - 매거진 제목
- 대표 영역 클릭 시 해당 매거진 페이지로 이동해야 한다.

### 5.1.2 이전 5개 매거진 캐러셀
- 최신 매거진 다음으로, **이전 발행 매거진 5개**를 carousel slide 형태로 보여줘야 한다.
- 각 카드에는 아래 정보가 포함되어야 한다.
  - 썸네일(Thumbnail)
  - 텍스트로 표시되는 호수
- 각 카드 클릭 시 해당 매거진 페이지로 이동해야 한다.
- 캐러셀은 좌우 이동이 가능해야 한다.
- 캐러셀 항목 수가 5개 미만일 경우 존재하는 매거진만 노출한다.

### 5.1.3 노출 기준
- 메인페이지에는 `published` 상태의 매거진만 노출된다.
- 최신 매거진은 `publishedAt` 기준 가장 최근 항목이어야 한다.
- 이전 5개 매거진은 최신 매거진을 제외한 나머지 `published` 매거진 중 최근 순으로 5개를 보여준다.

### 5.1.4 빈 상태 처리
- 발행된 매거진이 하나도 없을 경우 빈 상태 UI를 보여준다.
- 이 경우 최신 매거진 영역과 캐러셀 영역은 대체 문구 또는 placeholder UI로 표시한다.

### 5.1.5 메인페이지 반응형 요구사항
- 데스크톱에서는 최신 매거진 대표 영역과 지난호 캐러셀을 넓은 레이아웃으로 표시한다.
- 태블릿에서는 썸네일 크기와 여백을 축소하되 콘텐츠 구조는 유지한다.
- 모바일에서는 최신 매거진 대표 영역을 세로 중심 레이아웃으로 전환할 수 있어야 한다.
- 모바일 캐러셀은 터치 스와이프가 자연스럽게 동작해야 한다.
- 지난호 리스트의 썸네일과 호수 텍스트는 작은 화면에서도 식별 가능해야 한다.

## 6. 매거진 상세/뷰어 화면 요구사항

## 6.1 매거진 페이지 (`/magazines/:id`)
- 사용자가 최신 매거진 또는 지난호 매거진을 선택하면 매거진 페이지로 이동해야 한다.
- `published` 상태가 아닌 매거진은 일반 사용자 화면에서 접근할 수 없어야 한다.

## 6.2 뷰어 레이아웃
- 매거진 페이지는 **좌우 두 페이지(2-page spread)** 형태로 보여야 한다.
- 한 화면에 좌측 페이지 1장 + 우측 페이지 1장, 총 2장을 표시해야 한다.
- 첫 진입 시 첫 2페이지 세트가 보여야 한다.
- 페이지 수가 홀수일 경우 마지막 화면은 빈 페이지 또는 단일 페이지 보정 UI를 허용한다.

## 6.3 페이지 이동 방식
아래 방식으로 페이지 이동이 가능해야 한다.

- 좌/우 슬라이드 제스처
- 이전 페이지 버튼 클릭
- 다음 페이지 버튼 클릭

## 6.4 페이지 이동 규칙
- 다음 이동 시 2페이지 단위로 이동한다.
- 이전 이동 시 2페이지 단위로 이동한다.
- 첫 페이지 이전으로는 이동되지 않아야 한다.
- 마지막 페이지 이후로는 이동되지 않아야 한다.

## 6.5 페이지 렌더링
- 각 페이지는 업로드된 이미지로 렌더링한다.
- 각 페이지 이미지에는 관리자가 지정한 순서에 맞는 페이지 번호가 보여야 한다.
- 페이지 번호는 이미지 자체에 포함되어 있거나, 시스템 오버레이 방식으로 노출할 수 있다.
- MVP에서는 구현 단순화를 위해 **시스템이 페이지 번호를 UI 오버레이로 표시하는 방식**을 권장한다.

## 6.6 사용성 요구사항
- 페이지 이동 버튼은 사용자가 명확히 인지할 수 있어야 한다.
- 데스크톱 환경에서 책을 보는 느낌의 레이아웃을 제공해야 한다.
- 로딩 중에는 skeleton 또는 로딩 표시가 노출되어야 한다.

## 6.7 뷰어 반응형 요구사항
- 데스크톱에서는 좌우 2페이지 보기를 기본으로 제공해야 한다.
- 태블릿에서는 화면 너비에 따라 2페이지 보기 유지 여부를 결정할 수 있어야 한다.
- 모바일에서는 1페이지 보기로 자동 전환하는 방식을 기본 권장한다.
- 모바일에서는 스와이프 제스처로 다음/이전 페이지 이동이 쉬워야 한다.
- 페이지 이미지가 잘리지 않도록 화면 크기에 맞춰 비율 기반으로 축소 표시되어야 한다.
- 이전/다음 버튼은 작은 화면에서도 누르기 쉬운 크기여야 한다.

## 7. 관리자 화면 요구사항

## 7.1 관리자 진입 (`/admin`)
- 관리자는 `/admin` 경로로 접속할 수 있어야 한다.
- `/admin` 진입 시 매거진 목록 또는 관리자 대시보드로 연결되어야 한다.
- 관리자 화면 역시 반응형으로 구현되어야 한다.

## 7.2 매거진 목록
관리자는 매거진 목록에서 아래 정보를 확인할 수 있어야 한다.

- 매거진 호수
- 매거진 제목
- 대표 썸네일
- 상태 (`draft`, `published`, `unpublished`)
- 수정일시

관리자는 목록에서 아래 동작을 할 수 있어야 한다.

- 새 매거진 생성
- 기존 매거진 수정
- Publish 처리
- Unpublish 처리

## 7.3 매거진 생성/수정
관리자는 매거진 생성 및 수정 화면에서 아래 항목을 입력/관리할 수 있어야 한다.

### 7.3.1 기본 정보
- 매거진 번호 입력
- 매거진 제목 입력

### 7.3.2 페이지 이미지 업로드
- 매거진 콘텐츠는 이미지 단위로 업로드할 수 있어야 한다.
- 여러 장의 이미지를 순차적으로 업로드할 수 있어야 한다.
- 업로드된 이미지는 페이지 목록으로 표시되어야 한다.

### 7.3.3 페이지 번호
- 시스템은 업로드 순서 또는 정렬 순서에 따라 페이지 번호를 자동 부여해야 한다.
- 페이지 순서가 변경되면 페이지 번호도 자동으로 재정렬되어야 한다.

### 7.3.4 드래그앤드롭 정렬
- 관리자는 업로드된 페이지 이미지의 순서를 드래그앤드롭으로 변경할 수 있어야 한다.
- 순서 변경 후 저장 시 변경된 순서가 유지되어야 한다.

### 7.3.5 대표 썸네일
- 대표 썸네일은 아래 중 하나의 방식으로 설정할 수 있어야 한다.
  - 첫 페이지 이미지를 자동 대표 이미지로 사용
  - 별도의 표지 이미지를 대표 썸네일로 지정

> MVP 권장안: 첫 번째 페이지 이미지를 대표 썸네일로 자동 사용

## 7.3.6 관리자 화면 반응형 요구사항
- 데스크톱에서는 목록, 편집 폼, 이미지 리스트를 넓은 레이아웃으로 제공한다.
- 태블릿에서는 2단 또는 축소 레이아웃으로 자연스럽게 전환되어야 한다.
- 모바일에서는 관리자 기능 사용이 가능해야 하며, 폼과 이미지 리스트가 세로 스택 구조로 재배치되어야 한다.
- 드래그앤드롭이 어려운 작은 화면에서는 보조 정렬 UI(예: 위로/아래로 이동 버튼)를 허용할 수 있다.
- 업로드 버튼, 저장 버튼, Publish / Unpublish 버튼은 모바일에서도 쉽게 탭할 수 있어야 한다.

## 7.4 저장 및 상태 관리

### 7.4.1 드래프트 저장
- 관리자는 작성 중인 매거진을 드래프트로 저장할 수 있어야 한다.
- 드래프트 상태의 매거진은 일반 사용자 화면에 노출되지 않아야 한다.

### 7.4.2 Publish
- 관리자는 매거진을 Publish 할 수 있어야 한다.
- Publish 된 매거진은 일반 사용자 화면 메인/상세에서 접근 가능해야 한다.
- Publish 시점에 `publishedAt` 값이 기록되어야 한다.

### 7.4.3 Unpublish
- 관리자는 Publish 된 매거진을 Unpublish 할 수 있어야 한다.
- Unpublish 된 매거진은 일반 사용자 화면에서 즉시 숨겨져야 한다.

### 7.4.4 수정 정책
- Published 상태의 매거진도 수정 가능할 수 있다.
- 다만 MVP에서는 아래 중 하나를 선택한다.
  1. Published 매거진 직접 수정 허용
  2. 수정 후 재저장 시 즉시 반영

> MVP에서는 운영 단순화를 위해 **Published 매거진 직접 수정 후 즉시 반영** 정책을 권장한다.

---

## 8. 권한 및 접근 제어

## 8.1 일반 사용자
- `published` 상태의 매거진만 조회 가능

## 8.2 관리자
- 전체 매거진 조회 가능
- 드래프트/발행/비발행 상태 변경 가능
- 페이지 이미지 업로드 및 순서 변경 가능

## 8.3 접근 제한
- `/admin` 이하 경로는 인증된 관리자만 접근 가능해야 한다.
- 일반 사용자가 드래프트/비발행 매거진 상세 URL에 직접 접근해도 조회할 수 없어야 한다.

---

## 9. 화면별 상세 기능 명세

## 9.1 일반 사용자 메인페이지

## 10. API 요구사항 초안

## 10.1 일반 사용자 API
- `GET /api/magazines/latest`
  - 최신 발행 매거진 1개 조회
- `GET /api/magazines/recent?limit=5`
  - 최신 매거진 제외 이전 발행 매거진 조회
- `GET /api/magazines/:id`
  - 발행 상태의 매거진 상세 조회
- `GET /api/magazines/:id/pages`
  - 매거진 페이지 목록 조회

## 10.2 관리자 API
- `GET /api/admin/magazines`
  - 전체 매거진 목록 조회
- `POST /api/admin/magazines`
  - 매거진 생성
- `PUT /api/admin/magazines/:id`
  - 매거진 수정
- `POST /api/admin/magazines/:id/pages`
  - 페이지 이미지 업로드
- `PUT /api/admin/magazines/:id/pages/reorder`
  - 페이지 순서 변경
- `PUT /api/admin/magazines/:id/status`
  - 상태 변경 (`draft`, `published`, `unpublished`)

## 11. 유효성 검증 요구사항

## 11.1 매거진 생성/수정 시
- 매거진 번호는 필수값이어야 한다.
- 매거진 제목은 필수값이어야 한다.
- Publish 하려면 최소 1장 이상의 페이지 이미지가 필요하다.

## 11.2 이미지 업로드 시
- 허용 파일 형식: jpg, jpeg, png, webp
- 허용 용량 제한은 운영 정책에 따라 설정한다.
- 업로드 실패 시 사용자에게 오류 메시지를 보여줘야 한다.

## 12. 비기능 요구사항

## 12.1 성능
- 메인페이지는 최신 매거진과 지난호 리스트를 빠르게 렌더링해야 한다.
- 매거진 페이지는 첫 진입 시 초기 페이지 이미지를 우선 로드해야 한다.
- 다음 페이지 이미지는 프리로드를 고려할 수 있다.

## 12.2 반응형
- 모든 페이지는 모바일, 태블릿, 데스크톱 해상도를 고려해 반응형으로 구현되어야 한다.
- 일반 사용자 메인페이지, 매거진 뷰어, 관리자 목록, 관리자 편집 페이지 모두 반응형 대상에 포함된다.
- 데스크톱에서는 좌우 2페이지 뷰를 기본 제공한다.
- 태블릿에서는 화면 너비에 따라 2페이지 또는 1페이지 보기로 전환할 수 있어야 한다.
- 모바일에서는 1페이지 보기로 전환하는 방식을 기본 권장한다.
- 캐러셀, 버튼, 폼, 업로드 UI, 정렬 UI는 터치 중심 사용성을 고려해야 한다.
- 작은 화면에서도 텍스트 가독성과 조작 가능 영역이 유지되어야 한다.

> MVP 권장안: 모바일은 1페이지 보기로 전환

## 12.3 사용성
- 관리자 화면에서 업로드 상태와 저장 상태를 명확히 표시해야 한다.
- Publish / Unpublish 버튼은 현재 상태를 직관적으로 보여야 한다.

## 14. 개발 권장 구현 방향

## 14.1 프론트엔드
- 일반 사용자 화면과 관리자 화면은 동일 프로젝트 내 라우트 분리 방식 권장
- 예시:
  - Public: `/`, `/magazines/:id`
  - Admin: `/admin`, `/admin/magazines`, `/admin/magazines/:id/edit`

## 14.2 관리자 편집 방식
- 이미지 업로드 후 썸네일 리스트를 보여주고 드래그앤드롭으로 정렬
- 저장 시 sortOrder 기준으로 페이지 번호 자동 재계산

## 14.3 뷰어 구현 방식
- 페이지 단위 이미지 배열을 받아 책 형태 UI로 렌더링
- 데스크톱은 2페이지 spread
- 페이지 전환은 버튼 + 스와이프 지원

## 15. MVP 완료 기준 (Acceptance Criteria)

### 15.1 일반 사용자
- 메인페이지에서 최신 매거진 1개가 대표 썸네일과 함께 보인다.
- 메인페이지에서 최신 매거진 이전 5개가 캐러셀로 보인다.
- 각 매거진 클릭 시 해당 매거진 페이지로 이동한다.
- 매거진 페이지에서 좌우 2페이지 보기가 가능하다.
- 사용자는 슬라이드 또는 이전/다음 버튼으로 페이지를 넘길 수 있다.

### 15.2 관리자
- `/admin`에서 매거진 목록을 볼 수 있다.
- 관리자는 매거진 번호와 제목을 입력해 새 매거진을 만들 수 있다.
- 관리자는 매거진 페이지 이미지를 여러 장 업로드할 수 있다.
- 관리자는 페이지 이미지 순서를 드래그앤드롭으로 바꿀 수 있다.
- 관리자는 드래프트 저장을 할 수 있다.
- 관리자는 Publish / Unpublish 를 할 수 있다.

### 15.3 반응형
- 메인페이지는 모바일, 태블릿, 데스크톱에서 자연스럽게 레이아웃이 조정되어야 한다.
- 매거진 뷰어는 데스크톱 2페이지 보기와 모바일 1페이지 보기를 지원해야 한다.
- 관리자 화면은 작은 화면에서도 생성, 업로드, 저장, 발행 작업이 가능해야 한다.

- 데이터 모델 구성

## 17. 한 줄 요약
이 MVP는 **관리자가 이미지 기반 매거진을 업로드하고 발행하면, 일반 사용자는 메인페이지에서 최신/지난호를 탐색하고 책 형태의 2페이지 뷰어로 열람할 수 있는 구조**를 목표로 한다.
